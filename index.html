Групповые рекомендации на YAMBDA — как устроен весь пайплайн

Внутренний explainer для команды. Сквозной пример (группа из двух гостей — Аня и Боря, 4 трека) проходит через все стадии: от данных → обучение трансформера → таблица скоров → два агрегатора → метрики качества. Числа в стадиях 3–5 живые — можно править скоры и смотреть, как меняется всё ниже.

Чем ближе трек кому-то по звуку, тем весомее голос этого человека за этот трек. Задача агрегатора — научиться правильно расставлять эти веса: он подбирает их так, чтобы наверх итогового списка попадали треки, которые группа потом действительно слушает.

Для каждого трека: оценка × вес у Ани плюс оценка × вес у Бори = одно число, общая оценка трека. Вес = близость трека к человеку по звуку. Сортируем по этому числу — получаем список на двоих.

Главное: трек, который близок только одному из двоих, не теряется при простом усреднении — каждому достаётся своё. Ниже тот же путь разобран подробно: как трансформер взвешивает прослушивания и как агрегатор считает веса по звуку.

2

SASRec: обучаемый per-user скорерархитектура · loss · заморозка

▶

SASRec (Self-Attentive Sequential Recommendation) — трансформер, который по истории юзера предсказывает, какой трек он послушает следующим. На выходе — скор по каждому треку каталога. Это персональная модель: про группы она ничего не знает.

Архитектура

Вход: последовательность ID-треков [t₃, t₇, t₁, t₉, …] (последние ≤200)
↓
Item-эмбеддинги + позиционные эмбеддинги — каждый трек → вектор размерности 64, плюс позиция в последовательности
↓
2 × Transformer-блок (causal self-attention, 2 головы):
  • Self-Attention с causal-маской — позиция видит только прошлое (Q·Kᵀ, softmax, ·V)
  • Feed-Forward + residual + LayerNorm
↓
Репрезентация последней позиции → скалярное произведение со всеми item-эмбеддингами → скор по каждому треку

Главная идея self-attention: модель сама взвешивает, какие прошлые прослушивания важны для предсказания следующего. Causal-маска не даёт «подглядывать» в будущее.

Как учится (loss)

Для каждой позиции: правильный следующий трек (позитив) + 1 случайный негатив. Бинарная кросс-энтропия — позитив тянем к 1, негатив к 0:

BCE, n_neg = 1
где
  • — значение лосса (ошибки) на примере; обучение его минимизирует
  • — скор правильного следующего трека (позитив)
  • — скор случайного непрослушанного трека (негатив)
  • — сигмоида: превращает скор в вероятность 0…1

Агрегаторы: как из матрицы получить один топсначала методика, потом числа, потом формула

▶

Что именно мы считаем

На входе — матрица per_user [G × C] из стадии 3: строка на юзера, столбец на трек-кандидат. Задача агрегатора — свернуть эти строки в один групповой скор на каждый трек , по которому потом строится общий топ.

Методика у всех методов одна и та же — взвешенное голосование: каждому юзеру даётся вес, голоса (его персональные скоры) складываются. Различаются методы только тем, как назначается вес — фиксированный или зависящий от трека:

общая схема всех агрегаторов
где
  • — итоговый групповой скор трека i
  • — сумма по всем членам группы u (Аня, Боря, …)
  • — вес голоса человека u на треке i (насколько метод «доверяет» этому голосу)
  • — личный скор трека i для человека u (клетка матрицы из трансформера)
